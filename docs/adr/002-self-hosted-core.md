# ADR-002: 核心 Agent 内核自主实现，不直接依赖重型框架

- **日期**：2026-06-24
- **状态**：已采纳

## 背景

LangGraph/LangChain 等成熟框架功能全面但抽象层级高，学习曲线陡峭，定制化成本高，黑盒部分多不利于理解底层原理。

## 决策

核心 ReAct 循环、状态管理、工具协议等内核部分自主实现，保持 300 行左右极简代码；非核心通用能力按需引入轻量库；待系统复杂度足够高、自研成本超过框架收益时再考虑迁移到成熟框架。

## 多维度对比

### 核心架构层面

| 维度 | 当前自研实现 | 标准框架（LangGraph 等） |
|------|--------------|--------------------------|
| 执行范式 | 纯线性 ReAct 循环，单 Agent 单线程，无状态分支 | 支持状态图（StateGraph）编排，复杂分支、循环、条件跳转、并行节点 |
| 状态管理 | 仅维护原始消息列表，无结构化状态，无持久化 | 强类型结构化状态，自动状态合并、版本管理，支持 checkpoint 持久化 |
| 规划能力 | 纯反应式，走一步看一步，无预规划 | 内置规划器，复杂任务先分解为子任务 DAG |
| 记忆系统 | 无记忆，仅当前会话上下文，无压缩 | 分层记忆体系：短期工作记忆、长期语义记忆、情景记忆 |
| 反思纠错 | 无专门反思环节，仅靠工具错误提示被动修正 | 内置反思节点，支持进度评估、错误归因、策略调整 |

### 工具系统层面

| 维度 | 当前自研实现 | 标准框架 |
|------|--------------|----------|
| 工具生态 | 仅支持手动注册的本地 Python 函数 | 内置上百种预置工具，支持 MCP 协议接入外部工具 |
| 工具执行 | 本地同步执行，无隔离，无超时控制 | 支持沙箱隔离执行、异步执行、超时控制、重试机制 |
| 参数校验 | 基础类型校验 | 完整 JSON Schema 校验、参数自动补全、参数类型转换 |
| 工具选择 | 完全由模型自主选择 | 支持工具路由策略、权限控制、调用限流、成本优化选择 |

### 工程化能力层面

| 维度 | 当前自研实现 | 标准框架 |
|------|--------------|----------|
| 可观测性 | 仅基础事件推送，无统计 | 完整 Trace 链路追踪，Token 消耗、耗时、成本统计 |
| 错误处理 | 简单异常捕获，错误信息直接返回模型 | 分级错误处理、自动重试、降级策略、死循环检测 |
| 流式支持 | 仅 SSE 推送事件 | 支持 Token 级流式输出、工具调用过程流式 |
| 多模型支持 | 基础 Provider 切换 | 支持模型路由、fallback 机制、成本/质量动态权衡 |
| 评估体系 | 无 | 内置评估框架，支持 Golden Case 测试、A/B 测试 |
| 部署运维 | 单进程服务 | 支持分布式部署、任务队列、水平扩展、高可用 |

### 多 Agent 能力

| 维度 | 当前自研实现 | 标准框架 |
|------|--------------|----------|
| 多 Agent 协作 | 不支持 | 内置多种协作模式：层级式、对等协商、流水线、Swarm 模式 |
| 人机协作 | 仅支持手动停止 | 支持人工介入节点、审批流程、RLHF、Human-in-the-loop |
| 技能演化 | 无 | 支持从执行轨迹中自动学习优化工具使用策略 |

## 当前实现的核心优势

1. **极简透明**：核心代码仅 300 余行，没有任何黑盒，所有逻辑完全可控
2. **零依赖**：仅依赖 openai SDK 和 Python 标准库，部署简单
3. **性能 overhead 极低**：没有框架层的抽象损耗，执行速度快，调试方便
4. **灵活度高**：可以完全按照自己的需求定制

## 选型依据

1. 当前处于 Agent 开发早期阶段，核心目标是理解原理、快速迭代验证想法
2. 现有需求复杂度低，单 Agent ReAct 循环完全满足
3. 避免过早引入重型依赖导致的"框架绑架"，保留架构演进的灵活性
4. 核心逻辑自研可以更好地适配自定义需求

## 演进路线

1. **短期（当前）**：保持核心自研，逐步补充 P1 级能力
2. **中期**：当需要多 Agent 协作、复杂工作流时，按需引入 LangGraph 的部分组件
3. **长期**：如果系统复杂度增长到自研维护成本超过框架收益，再平滑迁移到成熟框架

## 后果

- 核心逻辑完全可控，便于学习原理和定制创新
- 无重型依赖 overhead
- 需要自行实现部分工程化能力，适合当前阶段需求
