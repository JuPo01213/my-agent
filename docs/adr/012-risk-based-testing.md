# ADR-012: 测试用例设计原则 — 风险点优先，不是覆盖率优先

- **日期**：2026-06-24
- **状态**：已采纳

## 背景

第 11 轮任务我写了 84 个单元测试并写成 ADR-011 庆祝"测试基线完成"，被用户直接批评："你写的这些测试有用吗？"。让子代理审查后得出结论：84 个里 **45 个低价值**（"为覆盖而覆盖"），主要是测 Python 标准库行为（dict 赋值、dataclass 默认值）而非项目逻辑。

## 用户原话

「把单元测试都删了你觉得你写的这些测试有用吗你再回过头去看看，或者让子代理审查一下呢 ADR-011这个是什么意思」

## 自我批评

1. **目标定错**：把"补基础"理解成"覆盖率优先"，但应该是"风险点优先"——每个测试都该回答"哪个未来改动会因为这个测试失败被打回？"
2. **没有"为什么"的论证**：84 个测试中只有 14 个能说清楚"守什么风险"，其余 70 个本质是 mock + 字典赋值的"伪测试"
3. **ADR-011 本身就是错的**：把"84 个测试 0.010s 全过"写成"测试基线完成"，没回答核心问题——"这 84 个测试在守什么？守得住吗？"

## 子代理审查结论

- 84 → 14 高价值 + 25 中价值 + **45 低价值**
- 关键问题：test_command.py 整体是"薄封装测试"、test_blackboard.py 17/24 是"测属性赋值"、test_parse_precondition.py 10/20 是"重复枚举"

## 决策

精简为 8 个真风险点守护测试（从 84 砍到 8，约 10%）：

- `tests/test_parse_precondition.py`：表驱动 1 个测试方法覆盖 4 类真风险（未知表达式放行 / 三 key and 真"全部满足" / malformed 表达式不崩 / snapshot 缺字段不崩）
- `tests/test_run_loop.py`：3 个测试（推理模型 fallback / 未知工具不崩 / max_steps 占位符）
- `tests/test_control_shell.py`：4 个测试（priority 排序 / 3 Agent 串行链 / 异常 failed / max_steps timeout 设计语义）
- **删除** `tests/test_command.py`（整体）+ `tests/test_blackboard.py`（整体）

## 测试设计新原则

每个测试都该回答这三个问题：

1. **守的是项目逻辑还是 Python 标准行为？** — 测标准行为即浪费
2. **未来哪个改动会因这个失败被打回？** — 没有答案的测试 = 无价值
3. **是否 mock 出来假行为？** — mock 仅在确认"代码自身反应正确"时有意义

## 修正 ADR-011

- 撤销"84 个测试"的统计
- 撤销"核心模块入口 + 关键分支 100% 有测试守护"的结论
- 真实状态：**8 个测试**覆盖**8 个真风险点**，不追求覆盖率

## 保留的真风险点（与 ADR 关联）

- `_parse_precondition` 三 key and 不退化（ADR-009 已修，此处守护）
- `run_loop` 推理模型 fallback（ADR-007 守护）
- `ControlShell` max_steps timeout 设计语义（ADR-011 文档化）
- `ControlShell` action 异常不崩
- priority 策略按 metadata 排序
- 3 Agent 串行链触发
- 未知工具返回错误字符串
- snapshot 缺字段容错

## 关联教训

1. **测试覆盖率是手段不是目标**：100% 覆盖率 + 全是低价值测试 = 0 价值
2. **子代理审查是必要的外部视角**：自己写的测试自己看，永远觉得有用
3. **ADR 不是庆功文**：ADR-011 写成"测试基线完成"是庆功文风，正确的 ADR 应该是承认决策风险 + 明确约束 + 留修订接口
4. **每个测试都需要"为什么存在"的注释**：本次精简后每个测试都有 "风险 N：xxx" docstring
5. **承认错误比掩盖错误更难能可贵**：ADR-012 本身就是对 ADR-011 结论的修订——项目允许 ADR 自我修订

## 当前测试统计（2026-06-24 第 12 轮完成时）

- 总数：**8 个测试**（从 84 砍到 8）
- 全部通过：true
- 平均执行时间：0.002s
- 覆盖真风险点：8 个
- 覆盖率：**不追求**（已放弃覆盖率指标）

## 后果

- tests/ 目录从 5 个文件缩到 3 个文件
- 每个测试都有明确的"风险 N"注释
- 后续 PR 改核心逻辑时，CI 跑这 8 个测试即可捕获大部分真问题
- 覆盖率指标从此不作为项目硬性要求
