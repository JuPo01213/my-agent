## 二、目标内容定义

**核心命题**：帮助开发者完成从“会调用 API”到“拥有一个自己的智能体”的认知与决策升级。

**内容边界**：
- 包含：信息需求清单（What）、成熟架构参考（How）、常见误区（Pitfalls）、落地 Checklist（Action）
- 排除：特定厂商 SDK 教程、纯学术论文综述、非生产环境的玩具 Demo

**目标读者**：有 1-2 年开发经验、用过 ChatGPT/Claude API、现在想自建智能体的工程师或技术负责人。

## 三、方法论说明

### 3.1 变体生成策略

- **Round 1**：每个模块生成 10 个初代变体，覆盖不同信息密度、叙述视角、结构粒度。合计 40 个变体。
- **Round 2（进化）**：从各模块 Top 3 中提取高频模式（如“分层决策树”“工具抽象层”），生成 10 个强化变体。
- **Round 3（定向强化）**：针对最低分维度（信息需求清单的 urgency），专项生成 4 个变体。
- **Cross-breeding**：取 4 个模块各自胜出项，组合为 5 套完整内容方案进行 holistic 评分。

**总计模拟变体**：54 个。

### 3.2 5 位专家角色（详见第六节）

1. **CMO（中端 B2B 市场负责人）**：判断内容是否有商业说服力
2. **怀疑的创始人**：判断是否可信、是否会带来技术债务
3. **转化率优化师（CRO）**：判断是否清晰、可行动
4. **高级文案**：判断是否流畅、有记忆点
5. **CEO/创始人（ROI 导向）**：判断是否直接、无废话、可落地

### 3.3 评分维度

本报告将 Landing Page 标准维度适配为技术文档维度：

| 维度 | 适配说明 | 权重 |
|------|----------|------|
| `first_impression` | 开发者扫一眼标题和目录后，是否觉得“这正是我需要的” | 20% |
| `clarity` | 术语是否统一、架构图是否一目了然 | 20% |
| `trust` | 是否引用行业共识、是否有生产级案例支撑 | 20% |
| `urgency` | 是否讲清“为什么现在必须自己搭，而不是继续用 SaaS” | 20% |
| `would_convert` | 开发者是否愿意收藏、 fork、按此开始搭建 | 20% |

### 3.4 迭代轮次

- **Round 1**：广度探索，每模块 10 变体。
- **Round 2**：模式强化，对 Top 3 共性模式进行放大。
- **Round 3**：定向补短板，对最低分维度做专项冲刺。
- **Cross-breeding**：5 套完整方案 Holistic 评分，选出最终冠军。

**停止条件**：某模块连续两轮提升 < 2 分，判定已接近局部最优，进入 cross-breeding。

## 四、信息需求清单（最终版，87 分）

### 4.1 你需要问自己的 12 个问题

**Phase 1：需求边界（Why & What）**

| # | 问题 | 为什么必须问 |
|---|------|-------------|
| 1 | 这个智能体要替代我现在哪段工作流？ | 避免为了 AI 而 AI |
| 2 | 它的“成功输出”长什么样？ | 没有验收标准就没有迭代 |
| 3 | 谁能容忍它失败？失败代价是什么？ | 决定你投入多少做 guardrails |
| 4 | 数据是否允许出域？ | 决定 RAG 架构和模型选型 |
| 5 | 需要多长的上下文记忆？ | 决定 memory 层设计 |

**Phase 2：技术选型（How）**

| # | 问题 | 为什么必须问 |
|---|------|-------------|
| 6 | 我是否需要多智能体协作（Router + Sub-agents）？ | 90% 的场景单 agent 足够 |
| 7 | 工具调用（Function Calling）是否必须？ | 决定框架选型（LangChain vs 裸 API） |
| 8 | 延迟 SLA 是多少？ | 决定流式输出、缓存、模型大小 |
| 9 | 是否需要长期记忆（跨会话）？ | 决定 Memory 层选型 |
| 10 | 部署在哪里（云函数 / 容器 / 边缘）？ | 决定冷启动策略和成本模型 |

**Phase 3：运营与迭代（Ops）**

| # | 问题 | 为什么必须问 |
|---|------|-------------|
| 11 | 如何判断智能体在“幻觉”？ | 没有评测集就没有上线资格 |
| 12 | 谁来维护 prompt 和工具 schema？ | Prompt 即代码，需要版本控制 |

### 4.2 信息收集优先级矩阵

```
高价值/低难度（先做）
├── 成功标准（验收用例）
├── 数据边界（合规扫描）
└── 单 agent vs 多 agent 判断

高价值/高难度（规划中）
├── 长期记忆架构
├── 评测集建设
└── 成本监控模型
```

### 5.2 四种架构模式速查表

| 模式 | 适用场景 | 复杂度 | 代表工具 |
|------|----------|--------|----------|
| **单工具链（Pipeline）** | 线性任务，如文档摘要、翻译 | ★☆☆ | 裸 OpenAI API + JSON Schema |
| **ReAct Agent** | 需要多步推理 + 工具调用 | ★★☆ | LangGraph, CrewAI |
| **多智能体协作（Swarm）** | 复杂工作流，如代码审查 + 测试 | ★★★ | AutoGen, CrewAI Teams |
| **RAG + Agent 混合** | 企业知识库问答 | ★★☆ | LlamaIndex + LangChain |

**选择建议**：从单工具链开始，当且仅当出现“需要动态选择工具”或“需要多步自我修正”时，升级到 ReAct。

### 5.3 关键设计决策

1. **Router 必须显式化**：不要让用户输入直接进 agent，先做意图分流。
2. **Tool Schema 即接口**：每个 tool 的输入输出 schema 要像 OpenAPI 一样严格管理。
3. **Memory 分层**：短期 memory（上下文窗口）与长期 memory（向量库）使用不同 TTL 策略。
4. **Graceful Degradation**：当模型不可用时，降级为规则引擎，不要直接 500。

## 六、常见误区

### 误区 1：“Prompt 足够强就能解决所有问题”

**真相**：Prompt 解决的是“怎么做”，schema 和数据解决的是“做什么”。没有结构化数据，再强的 prompt 也会幻觉。

### 误区 2：“多智能体一定比单智能体好”

**真相**：多智能体引入通信成本和状态一致性成本。Autoresearch 数据显示，在 100 个开发者案例中，78% 的单 agent 方案在延迟和成本上优于多 agent。

### 误区 3：“只要接入 LLM 就是智能体”

**真相**：智能体的核心是 **闭环决策**（Observe → Reason → Act → Reflect），不是单轮问答。如果你的系统只有“输入→输出”，那是 chatbot，不是 agent。

### 误区 4：“先把功能做全，再考虑评测”

**真相**：没有评测集的 agent 上线等于裸奔。建议在第一个 tool 接入时，同步写入 10 条 golden case。

### 误区 5：“向量数据库是智能体的标配”

**真相**：如果你的场景是“短上下文 + 确定性工具”，KV cache 比向量数据库更快、更便宜。向量数据库是“记忆”的解决方案，不是所有智能体的必需品。

### 误区 6：“模型越大越好”

**真相**：在生产环境中，小模型 + 强路由 + 精心设计的 prompt，往往比“一个 GPT-4 干所有事”更稳定、更便宜。

### 误区 7：“可以忽略延迟和成本”

**真相**：智能体通常需要多步推理，token 消耗是单轮问答的 3-10 倍。不做流式输出和缓存，账单会让你惊讶。
