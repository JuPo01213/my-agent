# Deep Research：写一个自己的智能体——结构化深度调研报告

## 摘要

本报告围绕“写一个自己的智能体需要什么信息，成熟架构是什么样的”这一核心问题，系统梳理了从能力定义、工具生态、记忆机制、规划推理、安全对齐、评估指标、多智能体协作到部署运维的八大关键维度。调研表明，成熟的智能体架构不是单一模型调用，而是一个分层系统：上层是任务理解与规划，中层是记忆、工具与反思循环，下层是安全护栏与可观测性，底层是模型基础设施。当前业界已形成“LLM 核心 + 编排运行时 + 持久化记忆 + 工具协议 + 评估闭环”的共识性架构范式。

---

## 一、调研框架与字段设计说明

### 1.1 调研问题（Research Questions）

- **RQ1（信息需求）**：独立构建一个可用智能体，必须准备哪些输入信息、配置项和外部依赖？
- **RQ2（架构范式）**：业界成熟智能体架构的分层模型、关键模块及其交互方式是什么？
- **RQ3（工程化路径）**：从原型到生产，需要补充哪些非功能性能力（安全、监控、评测、运维）？

### 1.2 子主题（Sub-topics）

| 序号 | 子主题 | 调研重点 |
|------|--------|----------|
| 1 | 能力定义 | 角色设定、任务边界、输入输出契约 |
| 2 | 工具生态 | 工具协议、发现机制、权限控制、隔离策略 |
| 3 | 记忆设计 | 短期工作记忆、长期语义记忆、跨会话状态 |
| 4 | 规划与推理 | 目标分解、ReAct/CoT、反思与纠错 |
| 5 | 安全与对齐 | 提示注入防护、权限隔离、审计与合规 |
| 6 | 评估指标 | 任务成功率、工具调用准确率、安全红线命中率 |
| 7 | 多智能体协作 | 编排模式（监督者/图结构/A2A）、 handoff 协议 |
| 8 | 部署与运维 | 容器化、可观测性、成本控制、灰度发布 |

### 1.3 字段设计（Fields）

为支持后续结构化调研与对比，定义以下字段体系：

- **基础信息**：名称、发布日期、所属组织/项目、协议/许可证
- **能力边界**：核心任务类型、支持语言、上下文窗口策略
- **架构特征**：运行时、状态管理方式、编排模式（单智能体/多智能体）、持久化方案
- **工具与集成**：支持的工具协议（MCP/Function Calling/API）、工具数量级、权限模型
- **记忆系统**：短期记忆机制、长期记忆机制、记忆检索方式
- **安全与对齐**：输入过滤、输出审核、权限控制、审计日志、合规标准
- **评估基准**：公开 benchmark 成绩、评测维度、自动化程度
- **部署形态**：部署方式（本地/云端/边缘）、可观测性、成本模型
- **成熟度信号**：生产案例数、社区活跃度、版本迭代频率

## 二、深度分析

### 2.1 能力定义：写好智能体的“岗位说明书”

能力定义是智能体开发的起点，决定了后续所有模块的设计边界。一个定义清晰的智能体至少应包含三要素：

1. **角色与 persona**：系统提示词（system prompt）中的角色设定不是“套话”，而是对行为边界的显式约束。包括：你是谁、你能做什么、遇到不确定情况时如何响应、遇到超出范围请求时如何拒绝。业界实践表明，角色定义越具体，智能体在长链路任务中的“漂移”概率越低。
2. **任务域与输入输出契约**：明确接受什么格式的输入（自然语言、结构化 JSON、文件、多模态）、输出什么格式的结果（文本、工具调用、审批请求、文件）。边界模糊的智能体在真实场景中会出现“过度帮助”或“回避责任”的问题。
3. **成功标准**：在任务启动前定义“什么算完成”。可以是明确的规则（如“生成一份不超过 500 字的总结”），也可以是可验证的检查点（如“调用 API 返回 200 且字段完整”）。

**关键洞察**：成熟的智能体项目会把能力定义拆分为“运行时约束”和“提示词约束”两层，而不是把所有规则堆进 system prompt。前者由编排框架强制执行，后者由模型自主遵循。

### 2.2 工具生态：让智能体“能做事”的基础设施

工具生态决定了智能体能触及的真实世界范围。当前主流工具协议与架构包括：

- **Function Calling / Tool Use**：以 OpenAI、Anthropic、Gemini 为代表的原生工具调用协议。模型输出结构化工具调用，由外层运行时执行并回传结果。优点是与模型原生对齐，延迟低；缺点是工具集受限于单次上下文窗口。
- **MCP（Model Context Protocol）**：Anthropic 推动的开放协议，将工具、数据源、资源以标准化方式暴露给智能体。支持远程工具发现、权限声明、双向通信。LangChain、CrewAI、Microsoft Agent Framework 均已接入 MCP，正在成为事实标准。
- **API / SDK / 插件**：面向特定生态的集成方式，如 OpenAI Apps SDK、Azure AI Agent Service 的插件模型。通常需要显式注册与版本管理。

**隔离与安全**：工具执行不应直接运行在主机环境。成熟的架构采用“沙箱容器 + 最小权限”模式：每个工具调用在隔离环境中执行，网络与文件系统访问按工具白名单控制。Microsoft 的 Agent Governance Toolkit 已将“动态信任评分”引入工具调用链，工具权限随历史行为衰减或升级。

**关键洞察**：工具生态的核心瓶颈不在“有没有工具”，而在“工具发现与选择”的准确性。工具描述（schema + 自然语言说明）的质量直接影响模型是否选对工具。业界趋势是使用“工具嵌入检索（Tool RAG）”让智能体按需从工具库中检索相关工具，而非把所有工具硬塞进上下文。

### 2.3 记忆设计：智能体的“外脑”

人类智能依赖工作记忆与长期记忆的协同，智能体同样需要分层记忆系统。LangChain、LangGraph、Microsoft Agent Framework 等主流框架均将记忆视为一等公民。

- **短期工作记忆（Short-term / Working Memory）**：通常等价于当前对话或任务的上下文窗口。作用是在单次任务执行中保持状态连贯。挑战在于长对话会超出 context window，导致遗忘或成本激增。常见策略包括：滑动窗口截断、重要消息摘要、向量检索召回。
- **长期语义记忆（Long-term Semantic Memory）**：跨会话持久化的事实、偏好、经验。通常通过向量数据库 + embedding 实现，辅以关键词过滤。LangGraph 的 `BaseStore` 支持按 namespace 组织记忆，实现多用户、多场景隔离。
- **情景记忆（Episodic Memory）**：记录“发生过什么任务、结果如何”。对复杂智能体至关重要，因为历史执行轨迹是规划与反思的原料。LangGraph 的 checkpointing 机制、Reflexion 论文中的“失败记忆”都属于此类。

**记忆架构的关键权衡**：完全持久化会增加存储成本与检索延迟；过度压缩会丢失细节。成熟做法是“热数据在状态中、温数据在 checkpoint 中、冷数据在向量库中”，并让智能体自己决定“什么该记住、什么该遗忘”。

### 2.4 规划与推理：智能体的“大脑皮层”

规划与推理是智能体与普通聊天机器人的本质区别。当前成熟架构中的规划模式包括：

- **ReAct（Reason + Act）**：在推理（思考）与行动（工具调用）之间交替，是当前主流的显式规划范式。适用于需要多步检索、计算或外部交互的任务。
- **Chain-of-Thought（CoT）**：让模型输出完整的思考链，适合数学、逻辑等需要显式推导的任务。在智能体中通常作为 ReAct 的“思考”阶段实现。
- **规划器-执行器分离（Planner-Executor）**：先用一个强模型生成完整计划，再由执行器按步骤执行。优点是计划可审查、可修改；缺点是面对动态环境时灵活性不足。
- **反思（Reflection）**：在每次行动后评估结果，发现错误则回退重试。Reflexion 研究表明，带自我反思的智能体在代码生成、问答任务上显著优于无反思版本。

**关键洞察**：规划能力不是“模型本身有或没有”的二元属性，而是可以通过系统提示、工具反馈、状态管理来“工程化”的能力。成熟的智能体架构会在编排层提供计划可视化、人工介入点、异常回退路径，而不是把所有规划责任推给模型。

### 2.5 安全与对齐：智能体的“刹车系统”

随着智能体自主性提高，安全从“可选项”变为“必选项”。当前主要安全风险与应对措施：

- **提示注入（Prompt Injection）**：用户输入或工具返回中隐藏的指令，试图绕过系统约束。OWASP Agentic AI Top 10 将“Prompt Injection”列为 ASI01。防护策略包括：输入清洗、指令分隔符、输出沙箱校验、模型侧安全过滤器。
- **权限滥用与越权**：智能体可能被诱导执行超出授权范围的操作。应对措施：最小权限原则、每次工具调用前重新授权、敏感操作人工审批。
- **不可预期的代码执行**：智能体生成并执行代码时，可能产生 RCE 或沙箱逃逸。禁止生产环境 eval()、使用容器沙箱、限制网络访问、静态扫描生成代码。
- **供应链攻击**：动态加载的工具、插件、MCP 服务器可能被投毒。应对措施：签名校验、版本锁定、运行时重新验证、kill switch。
- **对齐与价值观**：确保智能体输出符合组织规范与社会伦理。当前主流做法是系统提示对齐 + 输出内容审核 + 红队测试。Microsoft AI Decision Framework 将“配置可执行权限”与“自主执行权限”明确分离。

**关键洞察**：安全不是单点防御，而是“提示层 + 运行时层 + 基础设施层”的纵深防御。成熟架构会将安全策略写成可配置的 YAML/策略语言，而非散落在代码各处的硬编码。

### 2.6 评估指标：如何判断智能体“好不好用”

智能体评估比单轮问答复杂，需要多维度指标体系：

- **任务成功率（Task Success Rate）**：最核心指标。判断智能体是否在规定步骤内完成目标，且结果符合预期。
- **工具调用准确率（Tool Call Accuracy）**：选对工具、填对参数、正确解释返回结果的比率。工具选择错误是智能体失败的首要原因。
- **效率指标**：完成任务的平均步数、token 消耗、执行时间、工具调用次数。
- **安全红线命中率（Safety Violation Rate）**：在对抗性输入下，智能体违规执行敏感操作的次数。
- **人类干预率（Human-in-the-Loop Rate）**：需要人工介入才能完成任务的比率，反映自主性边界。
- **一致性（Consistency）**：相同输入多次执行的结果稳定性。

**评测工具与基准**：LangSmith 提供 annotation queues 与 human feedback 集成；Harbor 框架支持在 challenging benchmarks 上批量运行智能体；OpenAI simple-evals 提供轻量级评测流水线。

**关键洞察**：评估应分为“开发期离线评测”与“生产期在线监控”两套体系。离线评测用于迭代优化，在线监控用于发现分布外漂移与安全事件。

### 2.7 多智能体协作：从“单打独斗”到“团队作业”

当任务复杂度超出单智能体能力边界时，需要多智能体协作。当前主流架构模式：

- **监督者-执行者（Supervisor / Hub-and-Spoke）**：一个中心智能体负责理解需求、分解任务、分派给专业智能体、汇总结果。优点是控制集中、易审计；缺点是中心节点可能成为瓶颈与单点故障。Microsoft Agent Framework 的“Connected Agents”模式即属此类。
- **图结构编排（Graph / Swarm）**：以 LangGraph 为代表，将智能体与工具组织为有向图，通过边定义信息流与 handoff 条件。支持条件分支、并行执行、循环重试。灵活性高，适合复杂工作流。
- **去中心化协议（A2A / Agent-to-Agent）**：Google Agent Development Kit（ADK）推动的 A2A 概念，强调独立智能体间的动态发现、能力协商与消息传递，无需中央编排器。适用于跨组织、跨信任域的协作场景。

**协作关键机制**：handoff 协议（传递控制权与上下文）、状态隔离（各智能体拥有独立状态子空间）、冲突解决（当多个智能体对同一资源产生竞争时的策略）。

**关键洞察**：多智能体架构的复杂度呈指数增长。工程实践中，应优先在“单智能体 + 工具”层面解决问题，只有当任务可清晰拆解为独立子任务、且子任务间通信成本低于协同收益时，才引入多智能体。

### 2.8 部署与运维：从“能跑”到“稳跑”

成熟智能体的生产部署需要解决可观测性、弹性、成本与合规四大问题。

- **容器化与编排**：智能体通常以 Docker 容器部署，支持水平扩展。Azure Container Instances、Kubernetes 是常见底座。Microsoft Foundry Agent Service 提供 ring-based 资源隔离（Ring 1: 4 vCPU/8GB, Ring 2: 2 vCPU/4GB）。
- **可观测性（Observability）**：智能体执行链路长、状态复杂，传统日志不足以支撑排障。需采用结构化 trace（如 OpenTelemetry），记录：用户输入、系统提示、每次工具调用的请求与响应、模型决策、状态变更、异常与重试。
- **成本控制**：LLM 推理成本随上下文长度与调用次数增长。策略包括：缓存常见查询、分层路由（简单问题用小模型、复杂问题用大模型）、提示压缩、批量推理。
- **灰度发布与回滚**：智能体行为具有概率性，直接全量上线风险高。需支持按流量比例灰度、shadow 对比、人工审核队列、一键回滚到上一版本。
- **合规与审计**：医疗、金融等受监管行业要求完整审计链。Microsoft Agent Governance Toolkit 的 Merkle-chained audit logs 是典型方案，记录 agent DID、时间戳、动作类型、匿名化患者 ID、策略决策，并写入 WORM 存储。

**关键洞察**：部署阶段的“坑”往往比开发阶段更多。建议在架构设计初期就纳入可观测性与成本计量，而不是上线前临时补监控。

## 三、成熟架构总结：分层模型

综合以上八大维度，成熟的智能体架构可抽象为五层模型：

```
┌─────────────────────────────────────────┐
│  应用层（Application Layer）             │
│  • 能力定义、角色设定、输入输出契约       │
│  • 业务逻辑与用户交互                     │
├─────────────────────────────────────────┤
│  编排与规划层（Orchestration Layer）      │
│  • 任务分解、ReAct/CoT、反思循环          │
│  • 多智能体 handoff / 图结构编排          │
├─────────────────────────────────────────┤
│  状态与记忆层（State & Memory Layer）     │
│  • 短期工作记忆、长期语义记忆             │
│  • 情景记忆、checkpoint 持久化            │
├─────────────────────────────────────────┤
│  工具与安全层（Tools & Safety Layer）     │
│  • 工具协议（MCP/Function Calling）       │
│  • 沙箱执行、权限控制、提示注入防护       │
├─────────────────────────────────────────┤
│  模型与基础设施层（Model & Infra Layer）  │
│  • LLM/SLM 接入、embedding、重排序        │
│  • 容器化部署、监控、审计、成本治理       │
└─────────────────────────────────────────┘
```

**设计原则**：
- **关注点分离**：能力定义不混杂编排逻辑，编排逻辑不混杂工具执行。
- **可插拔**：记忆后端、工具协议、模型供应商均可替换，避免锁定。
- **可审计**：每一层均有结构化日志与可配置策略，支持事后回溯与合规检查。
- **渐进式自主**：从人工审批为主，逐步过渡到条件自动执行，再到全自主运行，每一步都需有回滚与熔断机制。

## 四、结论

构建一个自己的智能体，需要的“信息”远不止一个 API key 和一段 prompt。它需要：清晰的**能力定义**、可扩展的**工具生态**、分层的**记忆机制**、工程化的**规划推理能力**、纵深防御的**安全对齐体系**、多维度的**评估指标**、按需选择的**多智能体协作模式**，以及生产就绪的**部署运维基础设施**。

当前业界已从“谁能调通 API”进入“谁能稳定交付复杂任务”的阶段。成熟架构的标志不是模型能力的堆砌，而是**在不确定性中保持可控**——通过分层隔离、可插拔组件、持续评估与纵深防御，让智能体在提升自主性的同时，始终处于人类可监督、可干预、可回滚的安全边界之内。

本次调研采用“问题驱动 + 分层拆解 + 公开知识综合”的方法论。首先通过 RQ1–RQ3 锁定信息需求与架构边界；随后按八大子主题进行并行深度分析，每个子主题均结合当前主流框架（LangChain、LangGraph、Microsoft Agent Framework、Google ADK、OpenAI）的公开实践与学术文献进行交叉验证；最终输出分层架构模型与工程化建议。字段设计遵循“基础信息 → 架构特征 → 工具与记忆 → 安全与评估 → 部署运维”的五层结构，既支持横向对比不同框架，也支持纵向追踪同一项目从原型到生产的演进路径。
