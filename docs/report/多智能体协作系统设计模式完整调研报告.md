## 一、为什么需要多 Agent 协作

### 1.1 单 Agent 的硬数据瓶颈

| 上下文大小 | 任务成功率 | 原因 |
|------------|-----------|------|
| < 5K tokens | 94% | 任务清晰，AI 保持专注 |
| 5-20K tokens | 78% | 上下文管理良好 |
| 20-50K tokens | 61% | AI 开始"遗忘"早期上下文 |
| 50K+ tokens | 43% | 输出不一致，注意力分散 |

> 数据来源：[openhelm.ai](https://www.openhelm.ai/blog/multi-agent-systems-coordination-patterns)

**任务数量与成功率的关系**（41 家公司数据）：
- 5 个任务时 82% 准确率
- 10 个任务时 61% 准确率
- 15 个任务时 43% 准确率

### 1.2 多 Agent 协作的实际收益

| 维度 | 单 Agent | 多 Agent | 提升 |
|------|---------|----------|------|
| 任务成功率 | 47% | 89% | +89% |
| 复杂任务处理 | 34% | 91% | +168% |
| 任务处理时间 | 8.2 分钟 | 4.7 分钟 | -43% |
| 故障定位准确率 | 70% | 90%+ | +29% |

## 二、核心概念厘清

### 2.1 三个相关概念

| 概念 | 英文 | 本质 |
|------|------|------|
| **子代理** | Sub-agent | 大任务拆成小任务，每个小任务由一个子 Agent 处理 |
| **代理团** | Agent Team | 一组 Agent 形成固定团队，分工明确 |
| **多 Agent 协作** | Multi-agent | 多个 Agent 一起工作，涵盖上述两者 |

### 2.2 关键认识

**这些概念是层级关系，不是替代关系**：

```
多 Agent 协作
    ├── 子代理调用（处理短任务）
    ├── 代理团协作（处理长任务）
    └── 各种调度模式（Supervisor/Swarm/Hierarchical）
```

**生产实践**：这些模式是**按需融合**的，不是非此即彼。

## 三、LangGraph 官方四种架构

来源：[LangGraph Multi-agent Concepts](https://langchain-ai.github.io/langgraph/agents/multi-agent/)

### 3.1 四种架构对比

| 架构 | 控制方式 | 通信方式 | 适用场景 |
|------|----------|----------|----------|
| **Network** | 无中心 | 任意 Agent 互相通信 | 头脑风暴、复杂决策 |
| **Supervisor** | 中央控制 | Agent ↔ Supervisor | 任务明确、需集中控制 |
| **Hierarchical** | 多层中央 | 多层 Supervisor 嵌套 | 大型复杂任务 |
| **Custom** | 部分确定 | 部分 Agent 决定下一步 | 业务流程复杂、有明确路径 |

### 3.2 Supervisor 模式（最常用）

**官方代码**（来源：[langgraph-supervisor](https://github.com/langchain-ai/langgraph-supervisor-py)）：

```python
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor

def book_hotel(hotel_name: str):
    """订酒店"""
    return f"Successfully booked a stay at {hotel_name}."

def book_flight(from_airport: str, to_airport: str):
    """订机票"""
    return f"Successfully booked a flight from {from_airport} to {to_airport}."

flight_assistant = create_react_agent(
    model="openai:gpt-4o",
    tools=[book_flight],
    prompt="You are a flight booking assistant",
    name="flight_assistant"
)

hotel_assistant = create_react_agent(
    model="openai:gpt-4o",
    tools=[book_hotel],
    prompt="You are a hotel booking assistant",
    name="hotel_assistant"
)

supervisor = create_supervisor(
    agents=[flight_assistant, hotel_assistant],
    model=ChatOpenAI(model="gpt-4o"),
    prompt=(
        "You manage a hotel booking assistant and a"
        "flight booking assistant. Assign work to them."
    )
).compile()
```

## 四、四种调度模式

### 4.1 四种调度对比

| 模式 | 特点 | 判断标准 |
|------|------|----------|
| **Parallel（并行）** | 多任务同时执行 | 任务**无数据依赖** |
| **Sequential（顺序）** | 任务按顺序执行 | 有数据依赖 |
| **Loop（循环）** | 重复执行到满足条件 | 需要迭代优化 |
| **Routing（路由）** | 条件分发到不同处理路径 | 条件分支 |

### 4.2 并行的判断标准（重点）

**两个条件都满足才是真并行**：
1. 任务之间**没有数据依赖**（A 不需要 B 的结果）
2. 每个任务**输入相同**或独立

## 五、模式融合：实际生产中的混合模式

**实际生产中，各种模式按需组合**。

### 5.1 具体案例：算汇率+算金额

**用户问题**："我有 5000 人民币，换成美元能换多少？"

**架构分析**：
```
用户问题
   │
   ↓
[Supervisor] 分析任务
   │
   ├── 任务 1：算汇率 ──→ [子代理] 调用汇率 API ─┐
   │                                              │
   └── 任务 2：算金额 ──→ [子代理] 调用计算器 ───┤
                                                  │
                                                [并行]
                                                  │
                                                  ↓
                                              汇总输出
```

**为什么这样**：
- 两个子任务**互相独立**（金额 = 人民币 × 汇率）
- 任务简单，用**子代理**处理
- 用**并行**节省时间

---

### 5.2 具体案例：写代码

**用户问题**："写一个计算斐波那契数列的函数"

**架构分析**：
```
用户问题
   │
   ↓
[Supervisor] 分析任务
   │
   ↓
[子代理 1] 代码生成 Agent
   │ - 用 LLM 写代码
   │ - 返回代码
   ↓
[子代理 2] 代码执行 Agent
   │ - 在沙箱里跑代码
   │ - 验证结果
   ↓
[子代理 3] 错误处理 Agent（如果出错）
   │ - 重试或修正
   ↓
输出最终代码 + 执行结果
```

**为什么这样**：
- 短任务链，用**子代理串行**
- 不需要复杂团队
- 顺序明确

---

### 5.3 具体案例：处理用户投诉

**用户问题**："我昨天买的手机屏幕碎了，要退货"

**架构分析**：
```
用户问题
   │
   ↓
[Supervisor] 客服主管
   │ 分析：需要查订单 + 处理退货
   │
   ├── 任务 1：查订单 ──→ [代理团] 订单处理组
   │                       │ - 查订单状态
   │                       │ - 验证购买时间
   │                       │ - 查退换货政策
   │
   ├── 任务 2：评估问题 ──→ [子代理] 质量评估
   │                       │ - 判断是否在保修期
   │
   └── 任务 3：生成方案 ──→ [代理团] 售后组
                          │ - 退款 Agent
                          │ - 换货 Agent
                          │ - 维修 Agent
                          │ - 客服 Agent
   ↓
输出最终处理方案
```

**为什么用代理团**：
- 订单处理需要多步查询（**代理团**）
- 售后方案有多种选择（**代理团**）
- 质量评估很简单（**子代理**）

---

### 5.4 具体案例：写工作日报

**用户问题**："帮我生成今天的工作日报"

**架构分析（流水线 + 反思）**：
```
[1] 子代理：查今天的工作记录
    ↓
[2] 子代理：查邮件
    ↓
[3] 子代理：查会议
    ↓
[4] 子代理：查项目进度
    │
   [并行] 1-4 同时执行
    │
    ↓
[5] 反思 Agent：内容是否合理？
    ↓
[6] 输出最终日报
```

**为什么这样**：
- 步骤 1-4 都是**独立任务**（查邮件不需要查会议）
- 用**并行**节省时间
- 步骤 5 是**反思**节点（质量验证）

---

### 5.5 具体案例：实时翻译

**用户问题**："把这段英文翻译成中文"

**重要**：这**不是真正的并行**，而是一条**流水线**。

**错误分析**（常见误解）：
```
英文原文
   │
   ├── 翻译 Agent（英→中）   ← 翻译
   │
   ├── 校对 Agent（检查语法）  ← 等翻译完成才能校对
   │
   └── 文化适配 Agent（本地化）← 等翻译完成才能适配
```

**正确的架构分析**：
```
英文原文
   ↓
[翻译 Agent] 输出中文
   ↓
[校对 Agent] 检查语法（依赖翻译结果）
   ↓
[文化适配 Agent] 本地化（依赖翻译结果）
   ↓
最终结果
```

**为什么不是并行**：
- 校对 Agent 需要**翻译的结果**才能校对
- 文化适配 Agent 需要**翻译的结果**才能适配
- 有明确的数据依赖

---

### 5.6 具体案例：智能旅行规划（融合模式）

**用户问题**："我想去日本旅游 5 天"

**架构分析**：
```
用户：想去日本旅游
    ↓
[1] 顶层 Supervisor
    │ 分析：需要规划路线+订机票+订酒店+攻略
    │
    ├── 子代理：查天气（短期任务）────┐
    │                                  │
    ├── 代理团：行程规划（长期）       │
    │   ├── 景点调研 Agent             │
    │   ├── 交通规划 Agent             │
    │   └── 美食推荐 Agent             │
    │                                  │
    ├── 代理团：预订管理（长期）       │   并行执行
    │   ├── 机票 Agent                 │
    │   └── 酒店 Agent                 │
    │                                  │
    └── 子代理：生成攻略（一次性）─────┘
    ↓
[3] 反思层
    "攻略是否合理？预算是否超出？"
    ↓
[4] 输出最终方案
```

**融合点**：
- **Supervisor** 做顶层调度
- **子代理** 处理短、临时任务
- **代理团** 处理长、需要协作的任务
- **并行** 处理独立任务
- **反思** 验证质量

## 七、模式选择决策框架

### 7.1 根据任务特征选择

| 任务特征 | 推荐模式 | 实际案例 |
|----------|----------|----------|
| 任务独立、短 | **子代理 + 并行** | 算汇率+金额 |
| 顺序明确 | **子代理串行** | 写代码 |
| 业务复杂 | **Supervisor + 代理团** | 客服投诉 |
| 步骤固定、有反思 | **流水线 + 反思** | 写日报 |
| 步骤依赖、不可并行 | **流水线** | 翻译校对 |
| 大型综合任务 | **融合模式** | 旅行规划 |

### 7.2 根据团队规模选择

| Agent 数量 | 推荐模式 |
|-----------|----------|
| 1-3 个 | 子代理或简单 Supervisor |
| 3-8 个 | Supervisor + 子代理 |
| 8-15 个 | Hierarchical + Swarm |
| 15+ 个 | Hierarchical + 多个 Supervisor |

> **注意**：实际生产中 5-15 个 Agent 是最佳范围，超过 20 个协调开销会超过效率收益。

## 八、关键设计原则

### 8.1 通用原则

1. **单一职责**：每个 Agent 只做一件事
2. **搜索优先**：路由决策时优先搜索，不依赖 LLM 记忆
3. **消息传递要清晰**：Agent 之间传递的消息要明确
4. **状态要持久化**：使用 checkpointer 保存对话状态
5. **错误处理**：考虑 Agent 执行失败的情况
6. **上下文隔离**：每个 Agent 只关注自己的上下文

### 8.2 性能优化

1. **控制 Agent 数量**：5-15 个是最佳范围
2. **上下文管理**：避免单个 Agent 上下文超过 50K tokens
3. **并行执行**：独立任务并行处理
4. **结构化输出**：使用结构化输出验证决策

## 九、常见误区

### 9.1 误区 1：翻译任务是并行的

**错误**：把"翻译 + 校对 + 文化适配"设计成并行

**正确**：它们有数据依赖，必须串行（流水线）

### 9.2 误区 2：Agent 越多越好

**数据**：超过 20 个 Agent，协调开销超过效率收益

**建议**：5-15 个 Agent 是最佳范围

### 9.3 误区 3：所有任务都适合多 Agent

**事实**：简单任务用单 Agent 更快、更便宜

**判断标准**：任务超过 5-8 个、上下文超过 20K tokens 时考虑多 Agent
