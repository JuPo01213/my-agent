# 架构决策记录（ADR）

## ADR-001: 采用原生Function Calling替代正则解析工具调用
- **日期**：2026-06-24
- **背景**：最初使用Prompt强制格式+正则解析Action，工具调用成功率仅约60%，格式脆弱易出错
- **决策**：替换为大模型原生Function Calling能力，自动生成工具Schema，结构化返回工具调用
- **后果**：工具调用成功率提升至95%以上，支持多工具并行调用，减少90%格式解析错误，代码更简洁

## ADR-002: 核心Agent内核自主实现，不直接依赖重型框架
- **日期**：2026-06-24
- **背景**：LangGraph/LangChain等成熟框架功能全面但抽象层级高，学习曲线陡峭，定制化成本高，黑盒部分多不利于理解底层原理
- **决策**：核心ReAct循环、状态管理、工具协议等内核部分自主实现，保持300行左右极简代码；非核心通用能力按需引入轻量库；待系统复杂度足够高、自研成本超过框架收益时再考虑迁移到成熟框架
- **多维度对比详情**：

  #### 一、核心架构层面的差异
  | 维度 | 当前自研实现 | 标准框架（LangGraph等） |
  |------|--------------|--------------------------|
  | **执行范式** | 纯线性 ReAct 循环，单 Agent 单线程，无状态分支 | 支持状态图（StateGraph）编排，可以定义复杂的分支、循环、条件跳转、并行节点，支持 Plan-and-Execute、多 Agent 协作等多种范式 |
  | **状态管理** | 仅维护原始消息列表，无结构化状态，无持久化 | 强类型结构化状态（Pydantic Model），自动状态合并、版本管理，支持 checkpoint 持久化、断点续跑、轨迹回溯 |
  | **规划能力** | 纯反应式，走一步看一步，无预规划 | 内置规划器，复杂任务先分解为子任务 DAG，支持动态重规划、任务优先级调度 |
  | **记忆系统** | 无记忆，仅当前会话上下文，无压缩 | 分层记忆体系：短期工作记忆（自动摘要压缩）、长期语义记忆（向量数据库）、情景记忆（历史轨迹），支持跨会话经验复用 |
  | **反思纠错** | 无专门反思环节，仅靠工具错误提示被动修正 | 内置反思节点，支持进度评估、错误归因、策略调整，部分框架支持自我对弈优化 |

  #### 二、工具系统层面的差异
  | 维度 | 当前自研实现 | 标准框架 |
  |------|--------------|----------|
  | **工具生态** | 仅支持手动注册的本地 Python 函数 | 内置上百种预置工具（搜索、代码执行、数据库、API、文件操作等），支持 MCP 协议接入外部工具，支持工具动态发现 |
  | **工具执行** | 本地同步执行，无隔离，无超时控制 | 支持沙箱隔离执行（Docker/E2B）、异步执行、超时控制、重试机制、熔断降级 |
  | **参数校验** | 基础类型校验 | 完整 JSON Schema 校验、参数自动补全、参数类型转换、工具调用重试策略 |
  | **工具选择** | 完全由模型自主选择 | 支持工具路由策略、权限控制、调用限流、成本优化选择 |

  #### 三、工程化能力层面的差异
  | 维度 | 当前自研实现 | 标准框架 |
  |------|--------------|----------|
  | **可观测性** | 仅基础事件推送，无统计 | 完整 Trace 链路追踪，每一步 Token 消耗、耗时、成本统计，执行轨迹可视化，支持 LangSmith/LangFuse 等观测平台 |
  | **错误处理** | 简单异常捕获，错误信息直接返回模型 | 分级错误处理、自动重试、降级策略、死循环检测、人工介入节点 |
  | **流式支持** | 仅 SSE 推送事件 | 支持 Token 级流式输出、工具调用过程流式、中间状态实时推送 |
  | **多模型支持** | 基础 Provider 切换 | 支持模型路由、 fallback 机制、不同任务用不同模型、成本/质量动态权衡 |
  | **评估体系** | 无 | 内置评估框架，支持 Golden Case 测试、成功率/步数/成本等指标自动统计、A/B 测试 |
  | **部署运维** | 单进程服务 | 支持分布式部署、任务队列、水平扩展、负载均衡、高可用 |

  #### 四、多 Agent 能力差异
  | 维度 | 当前自研实现 | 标准框架 |
  |------|--------------|----------|
  | **多 Agent 协作** | 不支持 | 内置多种协作模式：层级式（Orchestrator-Worker）、对等协商、流水线、Swarm 模式，支持 Agent 间通信、任务 handoff、角色分工 |
  | **人机协作** | 仅支持手动停止 | 支持人工介入节点、审批流程、人类反馈强化学习（RLHF）、人类在回路（Human-in-the-loop） |
  | **技能演化** | 无 | 支持从执行轨迹中自动学习优化工具使用策略、自动生成新技能、技能库管理 |

  #### 五、当前实现的核心优势
  1. **极简透明**：核心代码仅 300 余行，没有任何黑盒，所有逻辑完全可控，方便学习原理和定制修改
  2. **零依赖**：仅依赖 `openai` SDK 和 Python 标准库，没有重型框架的依赖地狱，部署简单
  3. **性能 overhead 极低**：没有框架层的抽象损耗，执行速度快，调试方便
  4. **灵活度高**：可以完全按照自己的需求定制，不需要适配框架的抽象范式
- **选型依据**：
  1. 当前处于Agent开发早期阶段，核心目标是理解原理、快速迭代验证想法，自研内核可控性更高
  2. 现有需求复杂度低，单Agent ReAct循环完全满足，不需要复杂工作流和多Agent能力
  3. 避免过早引入重型依赖导致的"框架绑架"，保留架构演进的灵活性
  4. 核心逻辑自研可以更好地适配自定义需求，不需要为框架的通用能力付出额外 overhead
- **演进路线**：
  1. 短期（当前）：保持核心自研，逐步补充P1级能力（结构化状态、基础记忆、简单反思、可观测性）
  2. 中期：当需要多Agent协作、复杂工作流、持久化断点续跑等能力时，按需引入LangGraph的部分组件，而非全量迁移
  3. 长期：如果系统复杂度增长到自研维护成本超过框架收益，再平滑迁移到成熟框架
- **后果**：核心逻辑完全可控，便于学习原理和定制创新，无重型依赖 overhead；需要自行实现部分工程化能力，适合当前阶段需求

## ADR-003: 无安全护栏设计
- **日期**：2026-06-24
- **背景**：用户明确要求不需要任何安全护栏和权限限制，自行承担所有风险
- **决策**：所有工具调用、代码执行均直接放行，不做沙箱隔离、权限校验、敏感内容过滤

## ADR-004: 路由决策使用搜索而非记忆
- **日期**：2026-06-24
- **背景**：多 Agent 协作中，调度器（Supervisor/Router）需要判断下一步行动，但 LLM 的知识有截止日期，容易产生幻觉
- **决策**：路由决策时，优先使用搜索技能获取实时准确信息，而不是依赖 LLM 的内部记忆
- **路由原则**：
  1. **信息不确定时 → 搜索**：如果需要判断某事是否为事实，先搜索再决定
  2. **知识可能过时 → 搜索**：涉及最新资讯、技术文档、政策法规，必须搜索
  3. **明确常识可复用**：纯粹的逻辑推理、数学计算可以用内部知识
- **示例**：
  ```python
  # 错误：凭记忆判断
  def router(state):
      if "Python" in state.task:
          return "code_agent"  # LLM 记忆可能过时

  # 正确：搜索后判断
  def router(state):
      # 先搜索 Python 最新情况
      search_result = web_search("Python 最新版本 2024")
      if "版本" in search_result:
          return "code_agent"
  ```
- **后果**：路由准确性更高，减少幻觉导致的错误决策，但会增加搜索调用次数

## ADR-005: 后端核心分层重构 - core/ 与 multi_agent/ 分离
- **日期**：2026-06-24
- **背景**：原 `react_agent.py` 410 行混了 4 类职责：
  1. 工具注册（`TOOLS`、`register_tool`）
  2. ReAct 核心循环
  3. 多 Agent 协作参数（`tools`、`system_prompt`）
  4. Dashboard 事件协议（`on_event`、`stop_event`、事件格式）

  加上 `dashboard.py` 引入的"前后端耦合"问题——前端探索期的 SSE 推送、停止事件机制被塞进了核心 ReAct 循环，污染了核心抽象。
- **决策**：按"职责分离 + 关注点分离"原则完全重构：
  - **`agent_core/core/`**：最底层，只做 ReAct 循环 + 工具注册表
    - `react_agent.py`（~100行）：纯 `run_loop()`，返回最终答案字符串
    - `tool_registry.py`：TOOLS + register_tool + build_openai_tools_schema + validate_tool_args
  - **`agent_core/multi_agent/`**：多 Agent 协作的公开 API 层
    - `agent_api.py`：包装 core.run_loop，提供 `run_react_agent(user_input, client, model, tools, system_prompt, max_steps)`
    - `tool_filter.py`：`filter_tools_schema(tools)` 根据工具名列表过滤 OpenAI Schema
    - `tool_caller.py`：`call_tool_safe(tool_name, tool_args)` 安全调用工具
  - **删除** `dashboard.py`（前端探索期不再维护，前后端解耦）
  - **删除** 所有 dashboard 相关代码：`on_event`、`stop_event`、事件协议（thought/action/observation/final）
- **违反原则 vs 收益**：
  - **违反向后兼容**：完全重构优于兼容改造（项目规则已明确）
  - **违反 KISS（短文件）**：拆成 5 个文件，单个文件更简单
  - **收益**：
    - 关注点分离：核心循环不再背负"事件协议"这种 UI 概念
    - 公开 API 聚拢：Supervisor 只需 import `agent_core.multi_agent`
    - 可测试性：纯函数 `run_loop` 易于 mock
    - 可演进：未来加新 Agent 类型只需在 multi_agent/ 加新文件，不动 core
- **后果**：
  - `dashboard.py` 不再维护，前端探索期所有样式测试通过 `agent_core/static/*.html` 静态预览
  - core/ 与 multi_agent/ 之间单向依赖：multi_agent → core，不反向
  - 后续 Supervisor / Hierarchical / Swarm 等模式都基于 `run_react_agent` 实现
  - 若未来要恢复 dashboard 集成，需要在 multi_agent/ 上层新建 `ui/` 层（而不是改 core）

## ADR-006: 引入关系驱动多 Agent 协作引擎
- **日期**：2026-06-24
- **背景**：单 Agent ReAct 循环在面对复杂任务（多步分析、协作生产）时存在天然瓶颈：上下文超载、工具与人设混用、无法拆分专业角色。需要在 ADR-002 自研内核的基础上，不引入 LangGraph 重型依赖，实现可配置的多 Agent 协作能力。
- **决策**：在 `multi_agent/` 层新增 `relationship.py`（~330行），实现**关系驱动**的多 Agent 协作引擎，借鉴 4 个业界成熟实践：
  - **Blackboard 模式**（callisphere.ai / Hearsay-II 1970）：共享状态 + Knowledge Sources + Control Shell
  - **LangGraph Command**（langchain-ai.github.io）：`Command(goto, update, terminate)` 路由原语
  - **CrewAI YAML**（crewai.com 官方）：`role / goal / backstory` 三元组 + agents.yaml / tasks.yaml 声明式
  - **SALLMA 目录化**（佛罗伦萨大学 SATrends 2025）：Knowledge/Operational 双层 + 目录驱动
- **核心抽象**：
  - `Blackboard`：共享状态（facts / open_questions / history / status / metadata）
  - `Command`：`goto + update + terminate` 路由原语
  - `KnowledgeSource`：Agent 单元（preconditions + action + role/goal/backstory）
  - `ControlShell`：OODA 循环调度器（first_match / priority / round_robin 三种 strategy）
  - `RelationshipEngine`：YAML 驱动的协作引擎 + `from_yaml()` 工厂
- **强约束机制**：
  - `_parse_precondition()`：**安全子集表达式**（不引入 eval()），仅支持 `True/False`、`facts.has('key')`、`open_questions`、and 组合
  - `_activated` 集合：Agent 一次性激活，避免 first_match 死循环
  - 重复激活需用 `Command.goto` 显式转交（用于反思/迭代场景）
- **YAML 驱动**：
  - `config/agents.yaml`：Agent 定义（role / goal / backstory / tools / max_iter / preconditions / system_prompt）
  - `config/relationships.yaml`：关系定义（priority / termination）
  - **修改 YAML 即可改变协作流程，零改 Python 代码**
- **违反原则 vs 收益**：
  - **违反 KISS（短文件）**：单文件 330 行
  - **收益**：
    - 完全零外部依赖（除 PyYAML + openai SDK），保留自研内核优势
    - 同时支持 3 种调度策略 + 自定义 done_when 终止条件
    - 演示代码 (`demos/relationship_demo.py`) 覆盖 3 种典型用法：Python API / YAML 驱动 / 自定义 action
- **后果**：
  - v1.0 仅支持串行/优先级调度，**不**支持并行 / Swarm / Hierarchical
  - 默认 action 调 `run_react_agent`（复用 core/ 循环），KnowledgeSource 仍然受 max_steps 限制
  - 下一步演进：v1.1 加并行执行、v2.0 加 Swarm/Hierarchical、v3.0 加自动反思与记忆

## ADR-007: ReAct 循环兼容推理模型（reasoning_content fallback）
- **日期**：2026-06-24
- **背景**：`run_loop()`（位于 `agent_core/core/react_agent.py:101-103`）原本只读 `response_msg.content` 字段作为 final_answer。StepFun `step-3.7-flash` 等推理模型（reasoning model）会先在内部完成 CoT 推理，再产出短 final answer；`content` 经常为空或极短（仅 17 字符级别），真实完整输出在 `reasoning_content` 字段里。
- **问题表现**：
  - 2026-06-24-relationship-real.log 中 researcher 走默认 ReAct 循环，8 步 search 占位调用后 `final_answer` 为空
  - 2026-06-24-scenario.py 第一版（绕过 ReAct 循环直接 chat.completions）虽然能用 reasoning_content，但违背了"ReAct 循环是项目核心"原则（用户明确指出这是"本末倒置"）
- **决策**：在 `run_loop` 调 LLM 之后，`reply_content = response_msg.content or ""` 后增加 fallback：
  ```python
  if not reply_content and getattr(response_msg, "reasoning_content", None):
      reply_content = response_msg.reasoning_content
  ```
  满足两个条件：
  1. `content` 为空
  2. `reasoning_content` 存在
  才做 fallback；正常模型的 `content` 优先（避免污染非推理模型）。
- **优势**：
  - ReAct 循环保持完整运行（用户原始诉求）
  - 推理模型的 reasoning 过程 + final answer 全部进入 Blackboard.facts
  - 3 段产出从原来 17 字跃升至 6865+7600+9866 字
  - 对非推理模型零影响（content 仍优先）
- **替代方案评估**：
  - **A. 改 system_prompt 强制 LLM 不调 reasoning**：StepFun reasoning model 不能关掉 reasoning，方案不可行
  - **B. 在 default action 工厂里 merge reasoning_content**：增加一层抽象、不如在 run_loop 源头修
  - **C. 提供开关参数 `use_reasoning_content: bool`**：留给未来，当前用最简单的 fallback 即可
- **后果**：
  - 修复后，2026-06-24-scenario.py 在 129.6s 内完成 3 Agent ReAct 循环，8/8 KPI 全过
  - ReAct 循环对 StepFun、Doubao-pro、Claude-3.7-Sonnet 等推理模型都生效
  - 推理模型的输出会包含"思考过程"+"答案"，对最终用户而言略冗余（可后续让前端展示折叠）

## ADR-008: YAML 缺省 tools 字段的隐性陷阱
- **日期**：2026-06-24
- **背景**：2026-06-24 第一场场景实验（[experiments/2026-06-24-scenario.py](../experiments/2026-06-24-scenario.py)）跑通后，KPI 6/6 全过，但**实际工具调用 0 次**。用户问"过程中有工具调用吗"，grep 日志确认 `Action:` / `Observation:` / `tool_calls` 全部 0 次。根因不在 run_loop，而在 `relationship.py` 的 YAML 解析路径。
- **代码路径**（[relationship.py:533](../../agent_core/multi_agent/relationship.py#L533)）：
  ```python
  tools=cfg.get("tools", [])    # YAML 没写 tools → 缺省 []
  ```
  之后 [L182](../../agent_core/multi_agent/relationship.py#L182) `self.tools = tools or []` + [L196-197](../../agent_core/multi_agent/relationship.py#L196-L197)：
  ```python
  if self.tools:
      prompt_parts.append(f"你可以使用工具：{', '.join(self.tools)}。")
  # self.tools=[] → 整段 if 跳过 → prompt 里根本不告诉 LLM 有工具
  ```
  接着 [L236](../../agent_core/multi_agent/relationship.py#L236) `tools=self.tools or None` 传给 OpenAI client → LLM 看到 `tools: null` → 不可能发起 tool_call。
- **问题表现**：
  - YAML 注释写"默认为 None = LLM 看到所有已注册工具"（[2026-06-24-scenario.yaml L13](../experiments/2026-06-24-scenario.yaml#L13)）但代码默认是 `[]` 不是 `None`，注释与实现不符
  - 3 个 agent 全部 `tools=[]` → LLM 完全不知道 calculator / search / get_time 存在
  - 用户 query 写"请调用 calculator" 但 LLM 看不到 calculator，不会调
- **修复**（实验层，未改 core）：在 YAML 显式列 `tools: [calculator, get_time]`，让 LLM 真正看到工具。第二场场景 [2026-06-24-tool-scenario.yaml](../experiments/2026-06-24-tool-scenario.yaml) 跑出 **8 次 LLM 调用 + 10 次工具调用**（calculator × 8 + get_time × 2），`finish_reason=tool_calls` 6 次。
- **决策**：**保留 core 默认行为 `[]`**（不引入隐式的"全部工具"魔法），但同时采取以下补救：
  1. 在 `experiments/2026-06-24-tool-scenario.yaml` 模板**显式列 `tools:`**，附强制注释
  2. 计划在 `RelationshipEngine.from_yaml()` 里加 `warning` 日志：检测到 `tools` 字段缺省时打印 ⚠️（下次重构时实施）
  3. 修订 `experiments/2026-06-24-scenario.yaml` 注释（移除错误的"默认为 None"说法）
- **替代方案评估**：
  - **A. 改默认 `tools=cfg.get("tools", None)`**（None = 全部已注册）：更符合"零配置可用"原则，但破坏了"显式优于隐式"的工程风格，且需要 prompt 文案配合
  - **B. 加运行时警告**：对 zero-config 友好，不破坏显式风格，但只能在 from_yaml 阶段检测，无法覆盖直接构造 KnowledgeSource 的代码路径
  - **C. 引入 `tools_default` 全局配置**：增加配置项复杂度，对 v1.0 收益不明确
- **后果**：
  - 用户每次写 YAML 必须显式列 `tools:`，无法"忘了写就拿到全部工具"
  - 第二场实验 35.03s 完成工具调用全链路验证
  - 未来重构 `from_yaml()` 时应同时实施警告日志（A 或 B 方案）
- **关联教训**：与 ADR-005（核心层职责分离）一致——**核心代码不替用户做"看起来友好"的隐式行为**，避免出现 YAML 注释与代码不一致的认知陷阱。
