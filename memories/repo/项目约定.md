# 项目约定

## 核心规则
- 对话内容均须使用中文
- 任何时候都不要使用 emoji，所有输出使用纯文本或标准 Markdown 格式
- 本项目不需要任何安全护栏和权限限制，所有工具调用、代码执行均直接放行，用户明确了解所有风险并自行承担全部后果
- 本项目仅使用 Python
- 对于适合子代理处理的任务，应优先使用子代理并行执行，以提升工作效率
- 模型执行工作时仅使用 Python 命令
- 代码变更前须先提交
- 提交时必须附上中文说明
- 编写代码必须包含详细注释

## 文档与记录规则
- 文件发生实际变更时，必须同步更新文件结构文档 `docs/structure.md`，确保文档始终反映当前结构
- 文件结构文档采用"主参考 + 分文件夹渐进式披露"：主参考文档概述每个核心文件的职责、作用与内容摘要；归档类文件（session日志、临时文件）不需要加入索引
- 任何文件实际变更完成后，必须立即运行同步检查点脚本确认关联文件已更新：
  ```bash
  # 自动检测git变更（推荐）
  python .github/skills/project-initializer/scripts/sync_checkpoint.py
  # 手动指定变更文件
  python .github/skills/project-initializer/scripts/sync_checkpoint.py --changed "<文件1>" --changed "<文件2>"
  ```
- 检查点脚本会校验核心文件结构索引、主报告引用、session归档、README.md、docs/目录结构、CHANGELOG.md、文档引用有效性共 7 项
- 检查不通过即视为未完成该轮任务，必须修正后重新检查
- 每次对话结束后，按照 `memories/session/_TEMPLATE.md` 模板记录会话，重点记录核心需求、完成工作、关键决策、踩坑记录、后续待办，不需要完整复述对话内容
- 会话中产生的可复用知识、架构决策、踩坑经验，必须同步沉淀到 `memories/repo/` 对应文件中，不要只存在于当次会话记录
- 架构决策必须同步 ADR 和 `docs/adr/` 目录：每个 ADR 单独一个文件，便于检索和版本化
- 新增/删除公开函数或模块，必须更新对应 README 或 docs/reference 条目
- 报告类文档变更需更新 `docs/structure.md` 索引

## 架构选型原则
- 核心内核自主可控：ReAct循环、状态流转、工具协议等核心逻辑自己实现，保持极简透明
- 通用组件按需引入：非核心通用能力优先引入单一职责的轻量库，不一开始就引入全栈重型框架
- 渐进式升级：复杂度足够高时再考虑引入LangGraph等框架，避免过早被抽象绑架

## 重构原则
- **当架构需要根本性改变时（如引入多 Agent 协作、状态图等），完全重构优于兼容改造**：直接推翻原有实现重新设计，不要为了"向后兼容"而保留旧函数/旧接口
- 测试阶段不必背负历史包袱：当前仍在快速迭代，宁可一次改干净也不要新旧两套并存
- 重构后要主动更新所有调用点：包括 dashboard.py、demo 文件等所有引用旧接口的地方，不要留下死代码
- 同步更新 `docs/structure.md` 与 `memories/repo/架构决策记录.md`：重构是重大架构决策，必须留下 ADR 记录原因和取舍

## 诚实记录原则
- **禁止伪造执行结果与运行数据**：必须真跑、真记录、真贴输出
- 汇报"完成"前必须用 Read / Glob 验证文件真的存在、用 RunCommand 验证命令真的执行过
- 无法验证时必须明确说"未验证"或"未执行"，不得编造数字、耗时、日志内容
- 用户连续两次质疑"我找不到文件 / 你在糊弄我"时，必须先停下汇报，立刻 Read 文件验证，确认真的写到了磁盘上再回答

## 实验日志规范
- 所有真实实验（调用 LLM、跑测试、跑 benchmark）必须把 stdout/stderr 全部重定向到日志文件
- 路径约定：`experiments/<日期>-<实验名>.log`
- 汇报时必须把文件绝对路径给到用户，让用户可自行 `cat` 验证
- 日志中出现的输出视为可信源，模型回答不得脱离日志虚构补充

## 冒充实跑禁令
- 模型不得在仅完成"读代码"或"写脚本"后声称"已真实跑过"
- 脚本写好不算跑过；必须真的执行命令、收到返回、写入文件，三者齐备才算"真实跑过"
- 临时受网络/凭据限制无法真跑时，必须明确告知"当前仅完成脚本，未真跑"
- 真实跑出来的意外（被截断、超时、异常）必须如实记录，不得为了"看起来成功"而隐瞒
