## 关键决策与结论
- **场景选择**：用"传统软件公司 AI 转型"这种**强依赖行业经验与结构化思维**的任务，正好是 LLM 强项
- **架构优化**：默认 `_default_action` 走 `run_react_agent` 多轮 ReAct 循环，无工具时浪费——改为直调 `client.chat.completions.create(max_tokens=2000)`，单步从 34s 降到 5-8s
- **StepFun 推理模型特性**：step-3.7-flash 是 reasoning model，content 字段为空，真实输出在 `reasoning_content`（含"理解用户需求"+"推理步骤"+"最终答案"全过程）。max_tokens 太小时被截断；调到 2000 后能完整输出

## 踩坑记录
- **问题 1**：第一次跑 max_tokens=600，所有 3 个 Agent 产出 0 字
  **根因**：StepFun step-3.7-flash 是推理模型，把 tokens 全用在 reasoning 上，content 字段被截断为空
  **解决方案**：把 max_tokens 调到 2000；同时检测 content 为空时 fallback 到 reasoning_content
- **问题 2**：用户首次跑时跳过（执行超时）
  **根因**：第一次实现用默认 ReAct 循环，单 Agent 34s × 3 ≈ 100s+
  **解决方案**：实现 `simple_llm_action` 工厂，绕过 run_react_agent，单次 LLM 调用即可
- **问题 3**：KPI 检查代码沿用旧版"用 source 字段判 agent"导致 FAIL
  **根因**：`_apply_command` 硬编码 source="command"，agent 名字实际在 key 字段
  **解决方案**：改用 `h.get("key") == f"result::{agent}"` 判断，修复后 6/6 全过
- **问题 4**：PowerShell `>` 重定向丢中间输出（已知 bug，沿用之前方案：用 Python `logging.FileHandler` 内部写文件）
