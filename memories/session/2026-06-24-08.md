## 踩坑记录
- **问题 1**：第一版用 `simple_llm_action` 绕过 ReAct 循环
  **用户反馈**："本末倒置，有问题应该解决问题"
  **解决**：删除工厂函数，回到默认 ReAct 循环；发现"工具调不动"的根因是 `run_loop` 不读 `reasoning_content`，在源头修

- **问题 2**：回退默认 ReAct 循环后每段产出只有 17 字
  **根因**：`run_loop` 取 `response_msg.content`，推理模型该字段几乎为空
  **解决**：在 `run_loop` 加 fallback：content 为空时取 `reasoning_content`。这是"解决问题"的正确姿势

- **问题 3**：修复后单 Agent 耗时 40-48s（vs 之前的 5-8s 绕过版）
  **本质**：ReAct 循环让 LLM 自主决定深度，推理模型在 ReAct prompt 下会"展开"出更详细的答案
  **接受**：总耗时 129.6s 是 ReAct 完整运行的合理代价，3 段产出从 17 字跃升至 6865+7600+9866 字，价值远大于时间成本

- **问题 4**：KPI 段未写入 log（KPI 段需要主脚本完整执行到末尾）
  **原因**：log size 28165 chars，但 Python logging 在主进程结束前 flush，文件可能被 PowerShell 截断
  **解决**：写 `2026-06-24-scenario-kpi.py` 从 log 反推 KPI（独立于主脚本）
