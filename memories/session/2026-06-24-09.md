## 关键决策与结论

### 决策 1：保留 core 默认 tools=[]，但加补救
- **理由**：与 ADR-005 一致，核心层不替用户做"看起来友好"的隐式行为
- **补救**：YAML 模板强制显式列 tools + from_yaml() 警告日志（下次重构时实施）+ 修订错误注释

### 决策 2：LoggingClient 不并入 core/
- **理由**：它是测试基础设施，不是生产路径；并入会污染 ADR-005 的"核心只做一件事"原则
- **替代**：保留在 `experiments/2026-06-24-tool-scenario.py` 内，按需 import

### 决策 3：outputs/ 是产物目录，不是代码
- **理由**：手工归档保证零修改、可作为完整原文证据
- **不**做：自动生成 outputs/ 的脚本（避免污染）

### 决策 4：走读文档加"实战修正记录"小节而非重写
- **理由**：走读骨架仍有用，只需标注实战中发现的字段不匹配
- **范围**：L221-222 history 字段（实际是 `source="command"` + `key="result::<name>"`）

## 踩坑记录

### 踩坑 1：PowerShell `>` 重定向只保留最后一行
- **根因**：PowerShell 已知行为，cmdlet 输出流处理差异
- **解决方案**：用 Python `logging.FileHandler(mode="w", encoding="utf-8")` 绕过

### 踩坑 2：第一次跑工具场景发现 0 次工具调用
- **根因**：YAML 没列 tools（默认空） + 注释错误地说"默认为 None"
- **修复**：显式列 `tools: [calculator, get_time]` + 修订注释

### 踩坑 3：KPI 段未写入第一场日志（之前对话已遇到）
- **根因**：Python logging 缓冲 + PowerShell 重定向组合
- **解决**：第二场脚本改用纯 Python logging 写入，KPI 段稳定输出

### 踩坑 4：写作中错误引用了 `agent_core.demos.relationship_demo.py` 的 L145-170
- **根因**：走读是基于 demos/ 写的，实际关系引擎已迁移到 `agent_core/multi_agent/relationship.py`
- **解决**：实战修正记录里标注行号差异

### 踩坑 5：PowerShell `Get-Content` 显示中文乱码
- **根因**：PowerShell 控制台默认编码与 UTF-8 文件不匹配
- **解决方案**：用 `python -c "open(..., encoding='utf-8').read()"` 读文件
