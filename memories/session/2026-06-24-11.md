## 关键决策与结论

### 决策 1：max_steps 触发条件（真跑调试发现）
- **初版假设**：单 Agent + 永不 terminate → status=timeout
- **实际行为**：单 Agent 激活后下一轮 0 候选 → `source is None` → break → status=solved
- **根因**：`ControlShell.run()` 的 `_select_source()` 会过滤掉已激活的 Agent；timeout 分支走 `for-else`，要求循环**正常耗尽**（不 break）
- **真实触发条件**：`Agent 数量 ≥ max_steps`，让每步都能找到非空 candidates
- **解**：测试用 6 Agent + max_steps=5（5 步全跑 + 1 步 break 是 no-candidates）
- **代码无需修改**：这是 ControlShell 的设计语义，单 Agent timeout 路径在生产中实际不可达

### 决策 2：测试构造细节
- **避免互相依赖的 preconditions**：若 `a 看 result::b, b 看 result::a` → 0 候选立刻 solved（`status='solved', facts=[], step=0`）
- **优先 always-True preconditions**：测试 timeout/solved 路径时用 `precond_str="True"`，让所有 Agent 始终可激活
- **共享 action 工厂**：用 `make_loop(name)` 闭包避免重复代码，每个 Agent 自增自己的 fact 计数

### 决策 3：emoji 修复范围
- **只改 sync_checkpoint.py**（脚本）
- **不改** session 历史里的 ✅ ❌ 等（项目规则 2026-06-24-02 才加，之前的不追溯）
- **不改** `.github/skills/tavily/SKILL.md` 等其他文件（外部引用，保持原样）

## 踩坑记录

### 踩坑 1：test_max_steps_timeout 初版失败（status='solved' 而非 'timeout'）
- **现象**：2 Agent ping-pong 测试期望 timeout，实际 solved
- **诊断**：写 `experiments/2026-06-24-diagnose-timeout.py` 单独跑，发现 `status='solved', facts=[], step=0`
- **根因**：互相依赖的 preconditions 初始黑板两个 key 都没有 → 0 候选
- **修复**：改用 always-True preconditions + 6 Agent
- **教训**：调试 ControlShell 行为时，preconditions 设计直接影响循环路径

### 踩坑 2：2 Agent 还是 timeout 不触发
- **现象**：6 Agent 设计前先用 2 Agent 测，仍然 solved
- **根因**：2 Agent 跑 2 步（各自被激活）→ 第 3 步 0 候选 → break
- **修复**：Agent 数 ≥ max_steps

### 踩坑 3：PowerShell 引号转义让诊断脚本无法用 `python -c`
- **现象**：`python -c "..."` 在 PowerShell 里被解析成 `cmdlets`，引号嵌套导致 SyntaxError
- **解决**：用 `Set-Content -Value $content` 写脚本到文件，再 `python file.py`
- **教训**：复杂 Python 片段调试用 `Set-Content` 写脚本比 `python -c` 可靠得多

### 踩坑 4：multi_replace_string_in_file 部分失败
- **现象**：第一次 16 个 replace 中有 1 个失败（字符串未匹配）
- **根因**：第一次 try 包含 emoji 的 multi-replace，文件被前 15 个 replace 改了一部分，导致第 16 个 anchor 字符串已变化
- **解决**：用 `python` 脚本查 `non_ascii_lines` 定位剩余位置，单独 `replace_string_in_file` 补齐

### 踩坑 5：docs/structure.md 缺 tests/ 段（session 10 误删）
- **现象**：sync_checkpoint 报"6 个核心文件"但 structure.md 里没有 tests/ 段
- **根因**：session 10 用 `replace_string_in_file` 时 oldString 选错，误删了实验归档列表和 tests/ 段
- **修复**：补 tests/ 段 + 5 个测试文件 + 84 个测试统计
- **教训**：replace 大量内容前必须先用 read_file 看上下文，宁可分多步也不要"一步到位"
